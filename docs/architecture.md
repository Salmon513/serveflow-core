# Shared Package Architecture

```
@serveflow/shared
  ├── src/           ← TypeScript source (authoritative)
  │   └── types/
  │       ├── health-response.ts  (HealthResponse)
  │       ├── api-response.ts     (ApiResponse<T>)
  │       ├── booking.ts          (Booking, BookingStatus, CreateBookingRequest)
  │       └── customer.ts         (Customer, CreateCustomerRequest)
  └── dist/          ← compiled output (generated, gitignored)
      ├── index.js
      ├── index.d.ts
      └── types/
          ├── *.js
          ├── *.d.ts
          └── *.d.ts.map
```

Framework-agnostic: no NestJS decorators, no React imports, no ORM annotations.
Pure TypeScript interfaces and type aliases only.

---

# TypeScript Project References

```
apps/backend/tsconfig.json
  └── references → packages/shared/tsconfig.build.json

packages/shared/tsconfig.build.json
  └── composite: true (allows being referenced)
  └── declaration: true
  └── declarationMap: true
  └── outDir: dist/
```

Backend `tsc --build` triggers:
1. Checks if shared is stale (via tsconfig.build.tsbuildinfo)
2. Recompiles shared if needed → produces dist/
3. Compiles backend using shared's declarations

Frontend does NOT use TS project references — Next.js/Turbopack handles compilation.
Shared resolved via workspace symlink + `types: dist/index.d.ts` in package.json.

---

# Build Order

```
pnpm build:shared   →  tsc -p tsconfig.build.json  →  dist/
pnpm build:backend  →  tsc --build                  →  dist/
pnpm build:frontend →  next build                   →  .next/
```

Root scripts:
```json
"build": "pnpm build:shared && pnpm build:backend && pnpm build:frontend"
"typecheck": "pnpm build:shared && pnpm -r typecheck"
```

# Tailwind v4 Setup

Tailwind v4 uses a PostCSS plugin (`@tailwindcss/postcss`) and requires only one CSS file entry:

```css
/* app/globals.css */
@import "tailwindcss";
```

No `tailwind.config.ts` required for default configuration.
Custom theme tokens added via `@theme` directive in CSS when needed.

# Known Environment Warning

`next build` and `next dev` emit: "Found lockfile missing swc dependencies, patching..."

Root cause: Next.js tries to patch `pnpm-lock.yaml` to explicitly list SWC native binaries, but
cannot reach `localhost:4873` (local Verdaccio registry configured globally on this machine).
SWC IS installed and operational — `@next+swc-linux-x64-gnu@16.2.6` is present in pnpm store.
Warning is non-fatal. Resolved when the machine's global registry config is corrected.

# Shared Package Access Pattern

Phase 5 onwards — compiled output:
- `packages/shared/package.json` → `"main": "dist/index.js"` + `"types": "dist/index.d.ts"`
- Backend resolves via Node16 module resolution through workspace symlink
- Frontend resolves via Bundler module resolution through workspace symlink
- Both read compiled `.d.ts` declarations from `dist/`
